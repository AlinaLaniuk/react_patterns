# Components folder

Антипаттерн, связанный с низкоуровневой архитектурой приложения. Код, который меняется вместе, находится далеко. Код, который меняется по отдельности, находится рядом. Это проблема, возникающая при использовании архитектуры без архитектуры: код, разделенный на попаочки components, store, hooks и т.д.

<img width="422" height="915" alt="image" src="https://github.com/user-attachments/assets/2c974e54-839e-494d-9f29-1357c8995e86" />

Например, на скрине выше видно, что селекторы лежат рядом все вместе, а компоненты - отдельно в своей папке. При этом очевидно, что TrackModal скорее связан с селектором trackFormSlice, чем с TableRow. Но чтобы посмотреть, что происходит с trackFormSlice, нужно перейти в другую папку, то есть чтобы понять всю логику, нужно прыгать из папки в папку по проекту.

**Проблемы Components folder**

1. Сложно ориентироваться в проекте.
2. Для решения задачи нужно понимать весь код проекта.
3. Хрупкость из-а невозможности локализовать изменения.

Избавление от этот антипаттерна выглядит так. Вот пример архитектуры, в которой сущности лежат рядом друг с другом по технической принадлежности:
