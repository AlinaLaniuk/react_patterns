Антипаттерн global state coupling означает связанность через глобальное состояние. Фактически, это проблема выноса всех состояний приложения в глобальное, например, redux.

Перенос состояния в state manager приводит к хаотическим, сложно анализируемым связям в приложении.

Визуально код выглядит хорошо: нет огромных компонентов, нет props drilling, все неплохо с оптимизацией, потому что компоненты подписываются только на то, что им нужно. Но проблема такого подхода заключается в том, что компоненты фактически через этот глобальный стейт сильно свзяаны друг с другом. Например, в каком-то компоненте таблицы может в хэндлере очищаться состояние формы из модалки, просто потому, что технически такая возможность есть, и отследить это крайне сложно.
Из-за выноса стейта в глобальный исчезает инкапсуляция, однонаправленность данных, и в некотором роде, если сравнивать, god component и props drilling может быть даже лучшим решением, чем global state coupling.

**Проблемы global state coupling**

1.  Визуально компоненты простые, но понять приложение целиком очень сложно, потому что компоненты слишком сильно связаны и какие угодно данные могут откуда угодно имзениться.
2.  Код невозможно переиспользовать.
3.  Код становится слишком хрупким. В стандартном реакте без редакса, даже с нещадным пропс дриллингом, код стабилен, так как поток данных однонаправленный, он не такой ломучий.

**Как исправить?**

1. Не использовать глобальное состояние вообще. Использовать local global state (локальное состояние, вынесенное за передлы реакта)
2. Использовать паттерн mediator
